# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A **static Webflow export** for IOI Oleochemical Industries Berhad. Pages are pre-rendered HTML, served as static files. There is **no build step** for the site itself — what's in `en/` and `cn/` is what's deployed.

Two layers sit on top:

1. **Client-side CMS hydration** (`cms-hydrate.js` + `cms-config.js`): at runtime, JS reads `data-cms-*` attributes on the static HTML and fills them with content fetched from the **Contentful Delivery API**. This is why pages look "empty" in templates but populate live.
2. **Form submissions** post to an **AWS Lambda** (`https://uudwr4nux0.execute-api.ap-southeast-1.amazonaws.com/APIStage/contactus`, region `ap-southeast-1`) which sends email. Recipients are routed by integer ID encoded in the JS (e.g. company "IOI Acidchem" → `toRecipient: 22`). The Lambda source is managed externally in AWS, not in this repo.

The legacy `form-webmailContacts.asp` action on every form is **bypassed** by JS — never trust the form `action` attribute as the real submission target.

## Static / Dynamic content boundary

When working with content (translations, edits, search-and-replace):

- **Static** = anything *not* inside an element with `data-cms`, `data-cms-template`, `data-cms-list`, `data-cms-href`, or `data-cms-src`, and not on an element with class `w-dyn-bind-empty`. Lives in HTML files. Editable here.
- **Dynamic** = everything inside those markers. Comes from Contentful at runtime. **Editing these in HTML has no effect** — the markup is replaced on page load. Edit Contentful instead.

`cms-hydrate.js` has a long header comment documenting every supported `data-cms-*` attribute (single-entry detail, collection lists, reference lists, sorting, ordering, filtering). Read it before touching CMS-bound markup.

## Locales

- `en/` — source of truth, English (Webflow export, dropped in by the client periodically)
- `cn/` — Simplified Chinese, **generated by `tools/i18n/build.mjs` — do not hand-edit**
- A hidden language switcher (`#lnkLangOpt`) is wired up but currently hidden by JS (`langOpt.style.display = 'none'`); revealing it surfaces the locale toggle.

When a fresh Webflow export comes in, it lands in `en/` and may include duplicate files like `en/product-finder (2).html` or `en/about-us/job-listings2.html`. These are **ignored** by the i18n pipeline (see `IGNORE_PATTERNS` in `tools/i18n/extract.mjs`).

## i18n translation pipeline (most-touched workflow)

```
en/*.html  ──extract.mjs──►  i18n/en/*.json (reference)
                             i18n/cn/*.json (translator worksheet, edit `cn` field)

en/*.html + i18n/cn/*.json  ──build.mjs──►  cn/*.html
```

Common commands:

```bash
cd tools/i18n
npm install                    # one-time, installs cheerio
node extract.mjs               # re-extract; preserves existing cn translations on (id, en) match
node build.mjs                 # rebuild cn/*.html for any worksheet with at least one cn filled
python3 ../../tools/i18n/tag-status.py   # apply macOS Finder color tags (red/yellow/green) by completion
```

**Walker symmetry is critical.** `extract.mjs` and `build.mjs` share identical skip rules (SKIP_TAGS, SKIP_EXACT brand list, SKIP_REGEX patterns) and identical depth-first walk order. If you change one, change the other or the sequential `tNN` IDs will desync and translations will land in wrong places. The build validates by counting strings; mismatches mean drift.

`build.mjs` automatically:
- Sets `<html lang="zh-Hans">`
- Rewrites internal links `../../en/...` → `../../cn/...`
- Flips `#lnkLangOpt` href back to en (so cn pages link to en, en pages link to cn)
- Skips files where every `cn` field is empty (no half-translated English-with-rewritten-links output)

See `tools/i18n/README.md` for the full workflow.

## Form-related conventions

Forms use **reCAPTCHA v3** (site key `6LeD3HMsAAAAAHCDu9iGC-YSptGeh_HuPjIx_CZS`). Submission flow:

```
form submit → e.preventDefault()
  → grecaptcha.execute() generates token
  → token written to #g-recaptcha-response
  → submitToAPI(e) posts JSON to Lambda
```

Recipient routing is done **client-side** in `js/job-listings.js` and `js/enquiry.js` based on the selected company. If you add a new company, update the routing in JS and confirm the Lambda has the matching recipient ID.

`async: false` in the AJAX call is intentional — switching to `async: true` was tried and reverted (caused reCAPTCHA timing issues).

## Naming gotchas

- The site uses **Finsweet Attributes** (`@finsweet/attributes-cmsfilter@1`, `cmsselect@1`) loaded via `<script>` tag *after* CMS hydration completes. The async script tag in HTML was removed and re-injection is handled in `cms-hydrate.js` `reinitFinsweet()` to avoid double-init races.
- `data-cms-href-prefix` plus `data-cms-href` builds dynamic URLs — see how nav `?proapplication=` links work for the encoding pattern (URLSearchParams-style `+` for spaces, `%26` for `&`).
- Brand names that must NOT be translated or substituted: `IOI Oleochemical`, `IOI Acidchem`, `IOI Esterchem`, `IOI Pan-Century`, and product names `Palmac`, `Palmsurf`, `Palmsabun`. These are also the SKIP_EXACT list in the i18n walker.
